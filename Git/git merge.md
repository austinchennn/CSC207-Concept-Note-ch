这份关于 `git merge` 的笔记将帮助你理清“分叉”之后如何“合流”的逻辑。

---

## 🏗️ Git Merge 核心笔记

### 1. 核心逻辑：吸星大法

`git merge` 的本质是：**将另一个分支的修改，合并到你当前所在的“主场”分支。**

* **方向：** `目标分支` -> **指向** -> `当前分支`
* **口诀：** 我（当前分支）吸收了你（目标分支）。

---

### 2. 操作两步走

假设你想把 `bugFix` 分支的代码合并到 `main` 分支：

1. **先回到接收方：**
`git checkout main`
2. **把对方拽过来：**
`git merge bugFix`

---

### 3. 两种合并模式

Git 会根据两个分支的历史关系，自动选择合并方式：

#### A. 快进合并 (Fast-forward)

* **场景：** 自从你切出 `bugFix` 后，`main` 分支没有任何新提交。
* **结果：** `main` 指针直接像“快进”一样跳到 `bugFix` 的位置。不会产生新的提交记录。

#### B. 三方合并 (Three-way Merge)

* **场景：** `main` 和 `bugFix` 各自都有了新的提交（产生了分叉）。
* **结果：** Git 会找两个分支的共同祖先，把两边的改动揉在一起，并自动生成一个新的 **Merge Commit**。
* *这就是你在 **Learn Git Branching** 里看到的那个“菱形”汇合点。*



---

### 4. 冲突处理 (Merge Conflicts)

如果两个分支修改了**同一个文件的同一行**，Git 会因为不知道该听谁的而罢工。

* **状态：** 终端会提示 `CONFLICT (content): Merge conflict in ...`。
* **解决：**
1. 手动打开冲突文件。
2. 找到 `<<<<<<<`, `=======`, `>>>>>>>` 标记的内容。
3. 删掉不想要的，留下想要的，并删掉标记。
4. `git add <file>` 标记冲突已解决。
5. `git commit` 完成合并。



---

### 💡 进阶小贴士

* **取消合并：** 如果合并到一半发现全是冲突，想撤销重来，输入 `git merge --abort`。
* **合并信息：** 虽然在游戏里不用 `-m`，但在实战中，Git 默认生成的合并消息通常是 `Merge branch 'bugFix'`。如果你想自定义，可以使用 `git merge bugFix -m "你的合并说明"`。
* **查看分支关系：** 合并前后，可以用 `git log --graph --oneline --all` 在终端看到漂亮的彩色分支树。

---

> **一句话总结：**
> 永远先 `checkout` 到你最终想留下的那个分支，再 `merge` 那个你想吸收的分支。
