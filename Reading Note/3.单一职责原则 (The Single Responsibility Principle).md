## 1. 核心误解与定义演进

### 1.1 常见的误解
* ❌ **错误观点**：很多程序员根据字面意思，想当然地认为 SRP 意味着“每个模块都应该只做一件事”。
* 💡 **澄清**：
  * “一个函数只完成一个功能”确实是一个重要的设计原则（在将大型函数重构为小函数时频繁使用），但这只是**底层实现细节**，并不是面向系统架构层面的 SRP。

### 1.2 历史定义
> **“任何一个软件模块都应该有且仅有一个被修改的原因。”**

* **修改的原因**：软件系统的修改是为了满足“用户”和“利益相关者”的需求。因此，这些人/群体就是修改的原因。

### 1.3 最终定义（基于“行为者”）
> **“任何一个软件模块都应该只对某一类行为者（Actor）负责。”**

* 👤 **行为者（Actor）**：指一个或多个有共同需求的人（即希望对系统进行相似变更的群体）。
* 📦 **软件模块（Module）的定义**：
  * 大多数情况下，最简单的定义是指一个**源代码文件**。
  * 在某些不使用文件组织代码的语言/环境中，指**一组紧密相关的函数和数据结构**。
* 🔗 **内聚性（Cohesion）**：代码与数据就是靠着与某一类行为者的相关性被组合在一起的，内聚性本身就隐含了 SRP。

## 2. 违反 SRP 的两大反面案例（症状）

### 症状 1：重复的假象（Accidental Duplication）
以工资管理程序中的 `Employee` 类为例，它包含三个函数：
1. `calculatePay()` —— 由财务部门制定，向 **CFO** 汇报。
2. `reportHours()` —— 由人力资源部门制定，向 **COO** 汇报。
3. `save()` —— 由数据库管理员（DBA）制定，向 **CTO** 汇报。

```
+---------------------------------------+
|               Employee                |
+---------------------------------------+
| + calculatePay()  --> 负责 CFO 团队   |
| + reportHours()   --> 负责 COO 团队   |
| + save()          --> 负责 CTO 团队   |
+---------------------------------------+
```

* ⚠️ **产生的问题**：这三个函数服务于三类完全不同的行为者，却被强凑在同一个类中，导致行为者之间的行为被**强制耦合**。
* 💥 **灾难场景**：
  * `calculatePay()` 和 `reportHours()` 共享了一个计算正常工作时数的公共算法 `regularHours()`。
  * CFO 团队要求修改正常工作时数的计算方式，而 COO 团队（HR）不需要。
  * 负责修改的程序员注意到了 `calculatePay()` 调用了 `regularHours()`，便按照要求进行了修改并上线。但该程序员**没有注意到** `reportHours()` 也调用了它。
  * 最终导致 HR 部门的数据全错，给公司造成了数百万美元的损失。
* 📌 **启示**：**必须将不同行为者所依赖的代码强行切分开。**

### 症状 2：代码合并（Merges）
* 当一个源代码文件中包含许多服务于不同行为者的方法时，该文件必然会经历高频、并发的修改。
* 💥 **灾难场景**：
  * CTO 团队的 DBA 决定对 `Employee` 数据库表结构进行简单修改。
  * 与此同时，COO 团队的 HR  clerk 决定修改工作时数报表的格式。
  * 两位来自不同团队的程序员同时修改 `Employee` 类，导致致命的代码冲突（Conflict）。
* ⚠️ **风险**：现代合并工具虽然强大，但依然无法完全避免风险。这次合并不仅可能让 CTO 和 COO 的新功能出错，甚至可能波及到原本正常的 CFO 功能。

## 3. 解决方案（Solutions）

为了解决上述耦合问题，核心思想是将**相关的函数划分到不同的类中**。书中提出了三种具体演进方案：

### 方案一：数据与函数分离（最彻底的切分）
* 🛠️ **设计方式**：创建一个不包含任何函数的、十分简单的纯数据结构类 `EmployeeData`。
* 📂 **多类拆分**：设计三个独立的类（如 `PayCalculator`、`HourReporter`、`EmployeeSaver`），它们共同使用 `EmployeeData` 类。 each 类只持有其特定功能所需的代码。
* 🌟 **优点**：三个类互相不可见，完全不存在互相依赖，杜绝了重复的假象。
* 📉 **缺点**：程序员现在需要实例化并追踪三个不同的类。

```
+------------------+      +------------------+      +------------------+
|  PayCalculator   |      |   HourReporter   |      |  EmployeeSaver   |
+--------+---------+      +--------+---------+      +--------+---------+
         |                         |                         |
         +------------+------------+-------------------------+
                      | (使用)
                      v
            +-------------------+
            |   EmployeeData    |
            +-------------------+
```

### 方案二：使用外观模式（Facade Pattern）
* 🛠️ **设计方式**：为了缓解方案一中需要管理多个类的问题，引入一个 `EmployeeFacade` 类。
* 🌟 **职责**：`EmployeeFacade` 类本身包含的代码极少，仅负责初始化这三个具体实现类，并将调用请求分发（Delegate）给它们。

```
            +-------------------+
            |  EmployeeFacade   |
            +---------+---------+
                      | (实例化与调用分发)
         +------------+------------+
         v                         v                         v
+------------------+      +------------------+      +------------------+
|  PayCalculator   |      |   HourReporter   |      |  EmployeeSaver   |
+--------+---------+      +--------+---------+      +--------+---------+
         |                         |                         |
         +------------+------------+-------------------------+
                      | (使用)
                      v
            +-------------------+
            |   EmployeeData    |
            +-------------------+
```

### 方案三：核心业务与外观结合
* 🛠️ **设计方式**：如果更倾向于将最重要的业务逻辑与数据保持在近处，可以把最核心的方法（如 `calculatePay`）留在原本的 `Employee` 类中。
* 🌟 **职责**：将 `Employee` 类作为其他没那么重要函数的“外观（Facade）”，由它去调用其他较小功能类（如 `HourReporter` 和 `EmployeeSaver`）。

> 💡 **关于“类中函数过少”的辩证看法**
> 读者可能会觉得这样拆分会导致每个类只包含一个函数。但实际开发中并非如此，无论是计算工资、生成报表还是保存数据都是复杂的业务，每个类内部依然可能包含大量的**私有函数（Private Methods）**。每个类都分别容纳了一组作用于相同作用域的函数族，它们在各自的作用域外是互不可见的。

## 4. 本章小结

* **单一职责原则（SRP）** 表面上讨论的是**函数和类**之间的关系。
* 但它作为一种设计思想，在更高级别的软件层面会以不同的形式再次出现：
  1. 📦 **在组件层面**：它演变为 **共同闭包原则（CCP, Common Closure Principle）**。
  2. 🏗️ **在软件架构层面**：它演变为用于奠定架构边界的 **变更轴心（Axis of Change）**。
