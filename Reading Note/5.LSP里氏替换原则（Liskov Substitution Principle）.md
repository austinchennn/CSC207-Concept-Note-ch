里氏替换原则（LSP, Liskov Substitution Principle）由 Barbara Liskov 在 1988 年提出，其核心思想是：**子类型必须能够替换掉它们的父类型，且程序的行为保持不变。**

> [!abstract] 定义
> **If for each object o1 of type S there is an object o2 of type T such that for all programs P defined in terms of T, the behavior of P is unchanged when o1 is substituted for o2 then S is a subtype of T.**
> 
> 如果对于每个类型是 S 的对象 o1 都存在一个类型为 T 的对象 o2，能使操作 T 类型的程序 P 在用 o2 替换 o1 时行为保持不变，我们就可以将 S 称为 T 的子类型




## 继承层面的 LSP（面向对象场景）

在面向对象编程中，LSP 主要用于指导**继承**的正确使用。通过正反两个经典案例可以清晰地看出区别：

|**案例**|**关系描述**|**是否符合 LSP**|**原因分析**|
|---|---|---|---|
|正向案例：<br><br>  <br><br>授权费计算|`Personal` 和 `Business` 继承自 `License` 父类|**符合**|调用方（Billing）不关心具体是哪电子类，直接调用 `calcFee()`，两者完全可互相替换。|
|反向案例：<br><br>  <br><br>正方形/长方形|`Square`（正方形）继承自 `Rectangle`（长方形）|**违反**|长方形的宽和高可以独立修改，但正方形必须同时修改。这打破了调用方对“长方形”行为的假设，导致面积计算等逻辑出错。|

**违反 LSP 的典型后遗症：**

一旦违反了可替换性，调用方（User）就不得不增加 `if/else` 或类型判断（如 `if object is Square`）来特殊处理。这导致调用方与具体的子类产生了强耦合。

## 架构层面的 LSP（系统与接口场景）

随着软件工程的发展，LSP 已经从单纯的“面向对象继承指导”演变为了**更广泛的架构设计原则**。它现在适用于任何**接口与实现**的场景，包括：

- Java 风格的接口与实现类。
    
- Ruby 风格的鸭子类型（共享方法签名）。
    
- **分布式架构中的 RESTful 接口与微服务。**
    

### 违反 LSP 的架构灾难（出租车调度案例）

- **初始架构**：系统对接了多家出租车公司，所有公司统一遵循一个 REST API 标准（如包含 `pickupAddress`, `pickupTime`, `destination` 字段）。
    
- **发生变故**：Acme 公司（因为某些原因）没按标准开发，把 `destination` 字段缩写成了 `dest`。
    
- **架构污染**：因为 Acme 的接口**无法直接替换**标准接口，架构师被迫在调用端打补丁：
    
    - _最差做法_：在代码里硬编码写 `if (uri.startsWith("acme.com"))`，留下无穷的 Bug 隐患和安全漏洞。
        
    - _妥协做法_：增加一个复杂的配置数据库，通过 URI 动态路由映射不同的请求格式。
        

## 本章总结

1. **不要被表面的“Is-A（是一个）”关系迷惑**：正方形在数学上是一个长方形，但在编程行为上不是。判断是否能继承，关键看**行为是否具备可替换性**。
    
2. **LSP 是架构纯洁性的卫士**：无论是在类级别还是系统级别，一旦破坏了接口的可替换性，系统架构就会被迫引入大量复杂的、用于“擦屁股”的特殊应对机制（如条件判断、适配组件），最终导致架构腐化。
