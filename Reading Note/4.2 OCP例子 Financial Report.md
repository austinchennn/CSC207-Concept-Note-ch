把架构落实到具体的包（Package）和文件夹结构是非常关键的一步。在整洁架构（Clean Architecture）中，**包的划分不仅仅是为了分类，更是为了在物理层面（编译时）强制执行“依赖规则”**。

如果包划分得不好，开发者很容易写出错误的 `import`，从而打破架构边界。

下面我们将这个架构映射到一个标准的 Java 项目包结构中。我们以 `com.app.finance` 作为根目录，按层级（Layers）**和**组件（Components）来进行包的划分。

### 目录结构视图
```

│   └── FinancialEntities.java
│

│   ├── FinancialReportRequest.java      <DS>

│   ├── FinancialReportPresenter.java    <I> (Output Boundary)

│   └── FinancialReportGenerator.java    (Interactor 实现)

├── adapters/     <-- 3. 接口适配器层：转换数据 (Controllers, Presenters, Gateways)

│
├── infrastructure/             <-- 4. 最外层：框架与驱动 (UI, DB, Web)
```

### ☕ 核心代码与完整的依赖注入实现

#### 1. Domain 层（最内层，没有任何外部依赖）

这一层绝对不能出现外层包的 import。

Java

```
package com.app.finance.domain;

public class FinancialEntities {
    private double revenue;
    private double cost;

    public FinancialEntities(double revenue, double cost) {
        this.revenue = revenue;
        this.cost = cost;
    }
    // getters 和核心业务计算规则...
    public void deduct(double amount) { 
	    if (isFrozen) { 
			    throw new IllegalStateException("账户已冻结"); 
			    // 业务规则在这里得到保护！ 
		} if (balance < amount) { 
			throw new IllegalArgumentException("余额不足"); 
		} 
		this.balance -= amount; 
	}
}
```

#### 2. UseCases 层（核心应用逻辑，只依赖 Domain）

这一层定义了所有边界接口（Interfaces）和数据结构（Data Structures）。

Java

```
package com.app.finance.usecases;

import com.app.finance.domain.FinancialEntities; 

// <DS> 输入数据结构
public class FinancialReportRequest { 
    // 参数字段... 
}

// <DS> 输出数据结构
public class FinancialReportResponse { 
    public FinancialReportResponse(FinancialEntities data) { /* ... */ }
}

// <I> 数据库边界接口
public interface FinancialDataGateway {
    FinancialEntities getFinancialData();
}

// <I> 输入边界接口 (被 Controller 使用)
public interface FinancialReportRequester {
    FinancialReportResponse request(FinancialReportRequest request);
}

// <I> 输出边界接口 (被 Controller 使用，去触发 Presenter)
public interface FinancialReportPresenter {
    void present(FinancialReportResponse response);
}

// ---------------------------------------------------------
// Interactor 实现类：只依赖 Gateway 接口获取数据
// ---------------------------------------------------------
public class FinancialReportGenerator implements FinancialReportRequester {
    
    private final FinancialDataGateway gateway; // 依赖抽象

    // 依赖注入：通过构造函数强制要求传入 Gateway 实现
    public FinancialReportGenerator(FinancialDataGateway gateway) {
        this.gateway = gateway;
    }

    @Override
    public FinancialReportResponse request(FinancialReportRequest request) {
        // 1. 通过接口拿数据（不知道具体是啥数据库）
        FinancialEntities data = gateway.getFinancialData();
        // 2. 算账逻辑...
        // 3. 返回纯数据结构 Response 给调用方 (Controller)
        return new FinancialReportResponse(data);
    }
}
```

#### 3. Adapters 层（适配器，依赖 UseCases 和 Domain）

**Controller：作为调度中心，依赖 Requester 和 Presenter 接口。**

Java

```
package com.app.finance.adapters.controller;

import com.app.finance.usecases.*;

public class FinancialReportController {
    // 依赖抽象接口
    private final FinancialReportRequester interactor; 
    private final FinancialReportPresenter presenter;  

    // 依赖注入：需要执行器和展示器才能工作
    public FinancialReportController(FinancialReportRequester interactor, 
                                     FinancialReportPresenter presenter) {
        this.interactor = interactor;
        this.presenter = presenter;
    }

    public void handleReportRequest() {
        FinancialReportRequest request = new FinancialReportRequest();
        
        // 执行业务逻辑并拿到结果 (控制流：向右再向左返回)
        FinancialReportResponse response = interactor.request(request);
        
        // 交给 Presenter 去展示 (控制流：向下)
        presenter.present(response);
    }
}
```

**Presenter：将 Response 转化为 View 可以直接使用的数据（ViewModel）。**

```
package com.app.finance.adapters.presenter;

import com.app.finance.usecases.FinancialReportPresenter;
import com.app.finance.usecases.FinancialReportResponse;

public class ScreenViewModel { 
	public String reportTitle; // 报表标题 
	public String formattedRevenue; // 格式化后的收入 (比如 "$1,000,000.00") 
	public String formattedCost; // 格式化后的成本 
	public String profitColorText; // 盈利颜色的指令 (比如 "red" 或 "green") 
}

public interface ScreenView {
    void display(ScreenViewModel viewModel);
}

public class ScreenPresenter implements FinancialReportPresenter {
    private final ScreenView view; // 依赖 UI 接口

    // 依赖注入：需要传入具体的 View 实现
    public ScreenPresenter(ScreenView view) {
        this.view = view;
    }

    @Override
    public void present(FinancialReportResponse response) {
        ScreenViewModel vm = new ScreenViewModel(); // 解析 response 组装 vm
        view.display(vm);
    }
}
```

**Mapper：实现数据库接口。**

```
package com.app.finance.adapters.gateway;

import com.app.finance.usecases.FinancialDataGateway;
import com.app.finance.domain.FinancialEntities;
import com.app.finance.infrastructure.database.FinancialDatabase;

public class FinancialDataMapper implements FinancialDataGateway {
    private final FinancialDatabase db; // 依赖具体的底层设施

    // 依赖注入
    public FinancialDataMapper(FinancialDatabase db) {
        this.db = db;
    }

    @Override
    public FinancialEntities getFinancialData() {
        return db.queryEntities(); // 将底层数据转化为 Domain 实体
    }
}
```

### 🌟 组装的全部：Main 启动类 (Dependency Injection Wiring)

所有这些用 `private final` 声明并在构造函数中要求传入的依赖，最终需要在程序启动时进行“接线”。

在现代开发中，这部分工作通常由 Spring Boot 的 `@Autowired` 容器自动完成，但如果我们要手动写出本质（也叫手工装配），它长这样：

Java

```
package com.app.finance;

import com.app.finance.infrastructure.database.FinancialDatabase;
import com.app.finance.infrastructure.ui.WebView;
import com.app.finance.adapters.gateway.FinancialDataMapper;
import com.app.finance.adapters.presenter.ScreenPresenter;
import com.app.finance.usecases.FinancialReportGenerator;
import com.app.finance.adapters.controller.FinancialReportController;

public class Main {
    public static void main(String[] args) {
        
        // ==========================================
        // 1. 初始化最外层的技术细节 (Infrastructure)
        // ==========================================
        FinancialDatabase database = new FinancialDatabase();
        WebView webView = new WebView();

        // ==========================================
        // 2. 初始化适配器并注入外层细节 (Adapters)
        // ==========================================
        FinancialDataMapper gateway = new FinancialDataMapper(database);
        ScreenPresenter presenter = new ScreenPresenter(webView);

        // ==========================================
        // 3. 初始化用例层并注入适配器 (Use Cases)
        // ==========================================
        // Generator 只要求一个 FinancialDataGateway 接口，我们把 mapper 塞给它
	    FinancialReportGenerator interactor = new;
        FinancialReportGenerator(gateway);

        // ==========================================
        // 4. 初始化控制器并注入用例和展示器 (Controller)
        // ==========================================
        FinancialReportController controller = new FinancialReportController(interactor, presenter);

        // ==========================================
        // 5. 系统启动，接收外部请求
        // ==========================================
        // 假设用户点击了网页上的"生成报表"按钮
        controller.handleReportRequest();
    }
}
```

通过这套组装逻辑，所有的依赖箭头都完美契合了架构图，Controller 成为了整个请求和响应流转的中枢，而且最核心的业务逻辑对具体的数据库和 UI 毫无察觉。
