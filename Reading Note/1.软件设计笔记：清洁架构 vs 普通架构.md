# 软件设计笔记：清洁架构 vs 普通架构

## 一、 核心哲学对比

### 1. 普通架构 (以数据库为中心)

- **设计思路**：先设计数据库表，再写业务逻辑（DAO -> Service -> Controller）。
    
- **核心痛点**：业务逻辑被夹在 UI 层和数据库层之间，像三明治的肉饼。一旦换数据库（如 MySQL 换 MongoDB）或换接入方式（如 Web 换 RPC），核心逻辑必须大改。
    
- **依赖方向**：高层依赖底层。
    

### 2. 清洁架构 (以业务实体为中心)

- **设计思路**：先设计业务逻辑（Domain），无论外部环境如何变化，业务规则是不变的。
    
- **依赖规则 (Dependency Rule)**：**源代码依赖方向只能向内。** 内部核心（Entities/Use Cases）不知道任何外部细节（DB/UI/Framework）。
    
- **关键点**：使用**依赖倒置（DIP）**，将数据库、UI、第三方库视为“插件”。
    

---

## 二、 架构模型图示

- **内层 (Entities)**：最稳定的业务规则（如：折扣计算逻辑）。
    
- **中层 (Use Cases)**：特定应用的业务流程（如：下单流程）。
    
- **外层 (Adapters/Frameworks)**：最易变化的细节（如：REST 接口、SQL 数据库、Redis 缓存）。

## 三、 具体案例：用户注册功能

### 场景描述

用户输入用户名和密码，系统验证通过后存入数据库并发送欢迎通知。

#### 1. 普通架构的写法（高度耦合）

逻辑通常平铺在 `Service` 中，直接调用具体的 `Repository` 和 `EmailSDK`。

Java

```
// Controller 直接调用 Service
public class UserService {
    // 直接依赖具体的持久化实现和第三方邮件SDK
    private SqlUserRepository userRepository = new SqlUserRepository();
    private SendGridEmailProvider emailProvider = new SendGridEmailProvider();

    public void register(String name, String pwd) {
        if (userRepository.exists(name)) { // 业务逻辑依赖了SQL查询
            throw new Error("User exists");
        }
        User user = new User(name, pwd);
        userRepository.save(user); // 耦合了具体数据库
        emailProvider.send(name, "Welcome!"); // 耦合了具体邮件商
    }
}
```

- **问题**：想给这个 `register` 写单元测试？你必须先配置好 SQL 数据库和邮件服务器。
    

---

#### 2. 清洁架构的写法（解耦/插件化）

我们将业务流程提炼为 **Use Case**，将数据库操作定义为 **Interface (接口)**。

- **领域层 (Domain/UseCase)**：
    

Java

```
// 定义业务接口（也叫 Port）
interface UserRepository {
    boolean exists(String name);
    void save(User user);
}

// 核心业务逻辑：不依赖任何具体框架
public class RegisterUserUseCase {
    private final UserRepository repository; // 构造函数注入接口

    public void execute(String name, String pwd) {
        // 纯业务逻辑
        if (repository.exists(name)) throw new BusinessException("User exists");
        repository.save(new User(name, pwd));
    }
}
```

- **基础设施层 (Infrastructure/Adapter)**：
    
    这里才实现真正的 SQL 逻辑。如果以后想换成 MongoDB，只需要写一个新的实现类，**核心 `RegisterUserUseCase` 的代码一行都不用改。**
    

Java

```
public class MySqlAdapter implements UserRepository {
    public void save(User user) { /* 具体的 SQL 操作 */ }
    public boolean exists(String name) { /* SELECT ... */ }
}
```

## 四、 优劣势总结

|**维度**|**普通架构 (平铺式/三层)**|**清洁架构 (Clean)**|
|---|---|---|
|**上手难度**|低，直觉开发，代码量少|高，需要理解抽象和依赖倒置|
|**测试便利性**|难，必须 Mock 整个数据库/网络|极易，只需针对纯 Java/Python 对象测试|
|**维护成本**|随时间线性增长（容易坏）|前期高，后期低（业务逻辑被保护）|
|**第三方依赖**|强耦合，被框架绑架|框架只是工具，随时可替换|
|**适用项目**|简单的 CRUD、短期项目、Demo|复杂业务、需要长期维护的企业级应用|

## 五、 结论：我该怎么选？

- 如果你在做 **“一次性”** 的外包项目或简单的增删改查，**普通架构**效率最高。
    
- 如果你在做 **“长线运营”** 的产品，或者业务逻辑逻辑非常复杂（如保险计算、电商调度），**清洁架构**能让你在三年后依然敢于重构代码。
