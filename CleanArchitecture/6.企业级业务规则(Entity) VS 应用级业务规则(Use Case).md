这个困惑太经典了！你是真正把整洁架构（Clean Architecture）看进去了才会问出这个问题。几乎所有初学整洁架构的人，都会在“把代码写在 Use Case 还是 Entity”这个问题上卡住。

Uncle Bob 的核心思想是：**业务逻辑其实分为两种。**

我们要把它们明确区分开：**企业级业务规则（Entity）** 和 **应用级业务规则（Use Case）**。

## 1. 核心定义

### 实体（Entity）：企业级业务规则

它封装的是**最核心、最纯粹的业务本质**。

**判断标准（纸笔测试）：** 如果现在停电了，没有电脑，没有你的这个软件，大家靠纸、笔和计算器手工办公，这条规则还成立吗？如果成立，它就属于 Entity。

- **特征：** 它通常有状态（属性）和改变状态的方法。它**绝对不包含**任何对数据库、网络、第三方的调用。
    

### 用例（Use Case）：应用级业务规则

它封装的是**软件的运作流程（Orchestration/编排）**。

**判断标准：** 这条规则是不是只因为我们现在有了一个“自动化软件系统”才存在的？

- **特征：** 它是系统的“流程大管家”。它负责从数据库把 Entity 捞出来，调用 Entity 的方法去处理数据，然后再把 Entity 存回数据库，最后发个邮件通知用户。
    

## 2. 经典案例：银行转账（Transfer Money）

我们来看看一段“转账”的业务逻辑是如何被拆分到这两个地方的：

### 应该写在 Entity (`Account` 类) 里的逻辑：

- **逻辑 1：** 账户余额不能变成负数。
    
- **逻辑 2：** 如果扣款金额大于余额，抛出“余额不足”异常。
    
- **逻辑 3：** 每次存款，给账户增加相应的金额。
    

**为什么？** 因为就算是在 100 年前的钱庄，掌柜的用算盘算账，这些规则（余额不能小于0）也是绝对成立的。这叫“企业级业务规则”。

### 应该写在 Use Case (`TransferMoneyInteractor` 类) 里的逻辑：

- **逻辑 1：** 验证用户的登录 Token 是否有效。
    
- **逻辑 2：** 通过 `AccountRepository`（网关）从数据库查出转出账户 A 和转入账户 B。
    
- **逻辑 3：** 调用 `A.withdraw(100)` 和 `B.deposit(100)`（调用实体的纯业务逻辑）。
    
- **逻辑 4：** 如果转账成功，调用 `EmailGateway` 给用户发一封“转账成功”的提醒邮件。
    
- **逻辑 5：** 将 A 和 B 的最新状态通过 Repository 保存回数据库。
    

**为什么？** 因为“发邮件”、“查数据库”、“验证 Token”这些动作，是只有在现代软件系统中才会发生的事情。如果没有你的软件，这些流程是不存在的。这就是“应用级业务规则”。

## 3. 黄金辨别指南（快速测试法）

当你对着一段代码犹豫不决时，问自己以下三个问题：

|**测试问题**|**如果回答是 "Yes"**|
|---|---|
|**1. 纯度测试：** 这段代码需要调用数据库、文件系统、API 或发送网络请求吗？|坚决放进 **Use Case**。Entity 必须是纯内存的，零外部依赖。|
|**2. 流程测试：** 这段代码是在做“调度”工作吗？（比如：先查A，再算B，再存C）|坚决放进 **Use Case**。Use Case 就是一个编排剧本。|
|**3. 纸笔测试：** 就算没有电脑和软件，这项规则依然是业务的核心铁律吗？|坚决放进 **Entity**。（比如：订单总价 = 商品总价 * 0.9 的 VIP 折扣计算）。|

### 常见的错误做法（贫血模型）

很多人为了省事，会把所有的计算逻辑全塞在 Use Case 里，把 Entity 写成只有 `get` 和 `set` 方法的空壳（这被称为**贫血领域模型 Anemic Domain Model**）。

- _错误示范（在 Use Case 里写）：_ `if (account.getBalance() >= 100) { account.setBalance(account.getBalance() - 100); }`
    
- _正确示范（在 Entity 里写）：_ `account.withdraw(100);`（Use Case 只负责调用，计算和校验交给 Entity 自己）。



- `entity` = 最内层（核心业务规则）
    
- `usecase` = 用例层（应用流程编排）
    
- `adapter` = 接口适配器层（Controller、Gateway 实现，负责内外数据转换）
    
- `view` = 最外层（UI/前端展现）
