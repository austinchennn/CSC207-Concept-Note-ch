在软件开发（尤其是你正在学习的 **CSC207** 课程）中，**用例 (Use Case)** 是一个非常核心的概念。简单来说，它描述了**谁**（角色）使用系统来完成**什么**（目标）的一个完整过程。

根据你的课程笔记和软件开发生命周期（SDLC），可以从以下几个维度来理解：
### 1. 核心定义

用例是从用户的角度出发，描述系统如何与外部实体（称为“参与者”或 Actor）进行交互，以达到一个特定的业务目标。

- **参与者 (Actor)**：不仅指人，也可以是其他系统、硬件设备或时间触发器。
    
- **交互 (Interaction)**：描述参与者发送了什么请求，系统做出了什么响应。
    

### 2. 用例在团队中的作用

在软件团队中，不同角色对用例的关注点有所不同：

- **项目经理 (Project Manager)**：负责识别用例并管理开发人员之间的日常沟通。
    
- **UX 设计师 (UX Designer)**：根据用例设计用户如何在屏幕间导航、处理嵌套菜单，以及完成每个用例所需的努力程度。
    
- **QA（质量保证）**：负责确保程序确实实现了规范（Specification）中定义的用例功能。
    
### 3. 用例的组成要素

一个完整的用例通常包含：

1. **名称**：通常是一个动词短语（例如：提款、注册课程、查看成绩单）。
    
2. **前置条件**：在执行此用例前必须满足的状态（例如：用户已登录）。
    
3. **主流程 (Basic Flow)**：最理想、最顺畅的操作步骤。
    
4. **异常流程 (Alternative/Exception Flow)**：当出现错误或特殊情况时的处理逻辑（例如：密码错误、余额不足）。
    

### 4. 在 SDLC 中的位置

在软件开发生命周期的**定义需求 (Define Requirements)** 阶段，团队会决定开发团队需要完成哪些用例。随后在**测试 (Testing)** 阶段，QA 会针对这些用例编写测试用例，确保产品能够正常工作。

### 💡 举个例子（以你的作业提交系统 MarkUs 为例）

- **用例名称**：提交 Java 作业。
    
- **参与者**：学生（你）。
    
- **主流程**：
    
    1. 学生登录 MarkUs。
        
    2. 学生选择对应的课程和作业。
        
    3. 学生上传 `.java` 文件。
        
    4. 学生运行自测 (self-tests) 确保代码无误。
        
    5. 系统显示提交成功。
        

理解用例有助于你编写出更符合用户需求的代码，这也就是为什么 Dijkstra 强调程序必须能够被证明是正确的——因为我们要确保在所有可能的交互路径（用例流程）下，程序都能给出正确的结果。

你目前在练习的“Hello World”其实就是一个最简单的用例：**参与者（开发者）运行程序 -> 系统输出消息**。
